### Project scope: Entra ID emulator for app developers
- **Date**: 2026-06-22
- **Context**: Project kickoff. Need a local IdP so MSAL-based apps can develop/test sign-in and token flows without a real Entra ID tenant.
- **Decision**: Build "Entra Local" — an emulator exposing MSAL-compatible OIDC/OAuth2 endpoints, a minimal Microsoft Graph read surface, and a portal for managing users and app registrations. Captured in `specs/global-spec.md`.
- **Rationale**: Removes cloud dependency; enables offline/CI testing; avoids polluting shared tenants.
- **Alternatives**: Use a real test tenant; use a generic OIDC server (e.g. Keycloak) — rejected for lack of Entra/MSAL endpoint parity.

### Tech stack: Node.js + TypeScript with React portal
- **Date**: 2026-06-22
- **Context**: Need to choose the implementation stack.
- **Decision**: Node.js + TypeScript for the server; React + TypeScript (Vite) for the portal; `jose` for JWT/JWKS; SQLite for persistence; `zod` for validation.
- **Rationale**: Strong OAuth/OIDC ecosystem, easy single-exe packaging (Node SEA/pkg), fast portal development.
- **Alternatives**: .NET (natural Entra fit), Go (small static binary), Python/FastAPI — viable but Node chosen for portal velocity and packaging.

### MVP protocol scope
- **Date**: 2026-06-22
- **Context**: Defining which auth flows the emulator must support.
- **Decision**: Support Authorization Code + PKCE, Client Credentials, Refresh Token, and Device Code, plus OIDC discovery, JWKS, userinfo, and logout. Exclude Implicit, ROPC, OBO, SAML, and WS-Fed from the MVP.
- **Rationale**: Covers the dominant modern MSAL scenarios while keeping MVP focused.
- **Alternatives**: Add legacy flows (Implicit/ROPC) or SAML/WS-Fed — deferred/out of scope.

### Persistence: SQLite file
- **Date**: 2026-06-22
- **Context**: Choosing how to store users, apps, tenants, signing keys, and tokens.
- **Decision**: SQLite file with a repository layer; deterministic seed data; signing keys persisted for stable `kid`/signatures.
- **Rationale**: Persists across restarts, zero external deps, easy to reset/seed for CI.
- **Alternatives**: In-memory only (ephemeral), JSON files (no transactions/queries), pluggable store — deferred.

### Single tenant for MVP
- **Date**: 2026-06-22
- **Context**: Deciding directory cardinality.
- **Decision**: One fixed tenant (default GUID `11111111-1111-1111-1111-111111111111`); accept `common`/`organizations`/`consumers` aliases routing to it. Designed so multi-tenant can be added later.
- **Rationale**: Simplest mental model for the MVP.
- **Alternatives**: Full multi-tenant — moved to roadmap.

### TLS: HTTPS by default with auto-generated self-signed cert
- **Date**: 2026-06-22
- **Context**: MSAL fetches discovery/JWKS over HTTP(S); real-world parity benefits from HTTPS.
- **Decision**: HTTPS by default using an auto-generated, persisted self-signed cert; allow custom cert/key override and an HTTP fallback toggle.
- **Rationale**: Closer to real Entra behavior and avoids mixed-content issues; stable fingerprint across restarts.
- **Alternatives**: HTTP-only via loopback exemption; HTTPS opt-in — rejected in favor of secure-by-default.

### Sign-in UX and portal access
- **Date**: 2026-06-22
- **Context**: Interactive sign-in for Authorization Code flow and access control for the admin portal.
- **Decision**: Account picker by default with optional password enforcement (`REQUIRE_PASSWORD`); admin portal is open/unauthenticated.
- **Rationale**: Optimizes for local dev speed while allowing more realistic password login when needed.
- **Alternatives**: Password-only login; admin-protected portal — available as future toggles.

### Packaging targets
- **Date**: 2026-06-22
- **Context**: How developers will run the emulator.
- **Decision**: Support `npm start`, a single self-contained executable, and a Docker container as first-class targets sharing one config/data model.
- **Rationale**: Meets developers where they are (source, binary, or container/CI).
- **Alternatives**: Single target only — rejected to maximize adoption.

### Roadmap & MVP cut
- **Date**: 2026-06-22

### Iteration 1 HTTP framework: Fastify
- **Date**: 2026-06-22
- **Context**: Open roadmap decision tagged to Feature #1. Needed an HTTP framework for the emulator's OIDC/STS, Graph, and admin surfaces, resolved before writing Iteration 1 specs.
- **Decision**: Fastify (TypeScript).
- **Rationale**: First-class TS types and JSON-schema validation align with our validation approach; `fastify.inject()` directly powers the in-process integration-test layer our conventions require; built-in pino logging; plugin/encapsulation model maps cleanly onto the identity / graph / admin route groups.
- **Alternatives**: Express — wider familiarity but weaker TS ergonomics, no built-in schema validation, and less ergonomic in-process testing.

### Iteration 1 single-origin port & path layout
- **Date**: 2026-06-22
- **Context**: Open roadmap decision tagged to Feature #1 (serve the portal on the API port or a separate port). Needed a concrete URL map that emulates Entra's separate STS (`login.microsoftonline.com`) and Graph (`graph.microsoft.com`) hosts on a single local origin.
- **Decision**: Single port/origin. Portal SPA at `/` (SPA fallback for unmatched non-API routes); OIDC/STS under `/{tenant}/...` where `{tenant}` is allowlisted to the tenant GUID plus `common`/`organizations`/`consumers` — discovery at `/{tenant}/v2.0/.well-known/openid-configuration`, JWKS at `/{tenant}/discovery/v2.0/keys`, and `/{tenant}/oauth2/v2.0/{authorize,token,logout,devicecode}`; minimal Graph under `/graph/v1.0/{me,users,groups}` with UserInfo at `/graph/oidc/userinfo`; admin REST API under `/admin/api/...`; health at `/health`. In development the Vite dev server proxies to the API for HMR. MSAL configures `authority = <origin>/{tenant}` and Graph base `<origin>/graph`.
- **Rationale**: One origin means one cert to trust, no CORS, and trivial Docker/single-exe distribution; reserved path prefixes plus an allowlisted tenant segment prevent SPA/API route collisions; the prefix split mirrors Entra's real STS-vs-Graph host separation.
- **Alternatives**: Separate ports for portal and API — cleaner dev separation but doubles cert-trust/CORS friction and complicates packaging; rejected.

### Iteration 1 persistence driver: node:sqlite
- **Date**: 2026-06-22
- **Context**: Open roadmap decision tagged to Feature #2 (`better-sqlite3` vs `node:sqlite`); also interacts with the Iteration 2 single-executable goal (#17).
- **Decision**: Use Node's built-in `node:sqlite` (RC stage). Sets a Node 22.5+ floor (target current LTS / Node 24). Supersedes the unspecified "SQLite" driver in the earlier tech-stack decision.
- **Rationale**: No native bindings → simpler installs and a much easier path to the single-executable target (#17), avoiding `better-sqlite3`'s native-binding packaging problem; built into the runtime; synchronous API keeps the repository layer simple. User-directed.
- **Alternatives**: `better-sqlite3` — mature/fast but native bindings complicate single-exe packaging; rejected in favor of packaging simplicity. The newer-runtime requirement of `node:sqlite` is accepted given its RC-stage maturity.

### Iteration 1 cross-cutting spec contracts (Batch A: features #1–#6)
- **Date**: 2026-06-22
- **Context**: Writing the Batch A Iteration-1 specs surfaced cross-cutting contract choices that affect every later feature and that adversarial review (Decker) flagged for consistency/MSAL-correctness.
- **Decision**:
  1. **Reserved-stub lockstep.** Feature #1 registers `501 Not Implemented` JSON stubs for *all* canonical OIDC/OAuth/UserInfo/Graph paths in the path map; each later feature replaces its stub with a real handler. So every advertised discovery endpoint always resolves to a registered route (never a bare `404`/SPA), and the discovery doc can advertise core endpoints in dependency order without producing advertised-`404`s.
  2. **`client_info` in delegated token responses.** The token endpoint returns `client_info = base64url({uid:oid, utid:tenantId})` for user-present flows (auth code, refresh) to give MSAL stable account identity / cache keys; omitted for app-only client-credentials. Auth-code e2e pins MSAL's default AAD `protocolMode` + `knownAuthorities`.
  3. **Concrete-GUID issuer for all tenant aliases.** Requests via `common`/`organizations`/`consumers`/GUID all return the same GUID-form `issuer` (`${origin}/${TENANT_ID}/v2.0`), which equals every token `iss`. (Real Entra `/common` returns a templated `{tenantid}` issuer; the concrete GUID is simpler and MSAL-valid for a single fixed tenant. Cross-platform MSAL.NET/Python validation is #13.)
  4. **UserInfo path resolution.** UserInfo lives at `/graph/oidc/userinfo` (locked single-origin decision), overriding the draft global-spec `/{tenant}/openid/userinfo`. Discovery advertises this.
  5. **Determinism additions.** Timestamps stored as epoch-seconds integers; injectable clock in the token service; refresh tokens + sessions stored hashed/opaque; private signing keys stored as PKCS8 plaintext (documented dev-tool tradeoff); a fixed test signing key may be seeded for byte-reproducible JWKS/signatures in CI.
- **Rationale**: Keeps the six specs internally consistent and aligned with the locked architecture, and pre-empts the most common real-MSAL compatibility footguns (issuer matching, account identity).
- **Alternatives**: Per-feature ad-hoc stubbing (rejected — produces advertised-404s and ordering hazards); omit `client_info` and rely on OIDC-mode `sub` identity (rejected — fragile silent-token cache behavior in default MSAL mode); templated `{tenantid}` issuer (rejected — unnecessary complexity for a single tenant).

### Iteration 1 cross-cutting spec contracts (Batch B: features #7–#14)
- **Date**: 2026-06-22
- **Context**: Writing the Batch B Iteration-1 specs surfaced cross-cutting contract choices (refresh-token security, app-only authorization, Graph/admin/portal/run-target conventions) that build on Batch A's locked contracts and must stay consistent across the later features.
- **Decision**:
  1. **Refresh-token rotation + reuse detection (#7).** Every redemption rotates (revoke presented token, issue new one, `rotated_from` chain) inside **one atomic transaction** (compare-and-set on `revoked`), so concurrent redemptions never double-mint. Replaying an already-revoked token triggers **whole-family revocation** along the `rotated_from` chain reachable from the presented token (separate sign-ins are separate chains, preserved) and returns `invalid_grant` — reuse takes precedence over expiry. `getByHash` must return revoked/expired rows. Rolling TTL (fresh `TOKEN_LIFETIME_REFRESH_SECONDS` per rotation), no absolute family cap in MVP. ID token re-minted on refresh when `openid` granted; PKCE not re-checked on refresh; new `refresh_token` returned only when `offline_access` is in the (post-narrowing) grant.
  2. **Client-credentials app-role auto-grant (#8).** No app-role *assignment* table in MVP. App-only `roles` = all enabled `app_roles` on the resolved **resource** app whose `allowed_member_types` includes `Application` (empty `[]` for Graph/unknown resource). `.default` resource resolves to Graph (`GRAPH_RESOURCE_ID`), a registered app's `app_id_uri` (now enforced **unique** by #11), or its GUID `app_id`; else `invalid_scope`. Secret-only client auth. Documented divergence from Entra (which requires explicit application-permission assignment + admin consent), consistent with the locked auto-consent model.
  3. **UserInfo + logout (#9).** UserInfo (`/graph/oidc/userinfo`) requires a **delegated** Graph-audience token (has `oid`); app-only → `403 insufficient_scope`. `sub` returned equals the token `sub`. Logout validates `post_logout_redirect_uri` by exact match against the resolved app's **registered redirect URIs** (no separate post-logout-URI registry in MVP) and requires a resolvable `client_id`; otherwise renders a signed-out page. `id_token_hint` signature not enforced (hint only). Front-channel multi-RP logout is out of scope, so its discovery capability flags are not advertised (lockstep).
  4. **Minimal Graph authorization + Graph delegated-scope carve-out (#10, amends #5/#6).** Graph accepts the emulator's own `ver:"2.0"`, `aud=GRAPH_RESOURCE_ID` tokens (never Microsoft v1 shape). `/me` requires a delegated token; other reads accept delegated or app-only. **No fine-grained Graph scope/role enforcement in MVP** (a valid Graph-audience token suffices). **Amendment to Batch A #5/#6:** Microsoft Graph *delegated* scopes — the fully-qualified `<GRAPH_RESOURCE_ID>/<name>` form or a recognized bare short-name (`User.Read`, `User.Read.All`, `Group.Read.All`, ...) — are **auto-consented and accepted without registration**, set `aud=GRAPH_RESOURCE_ID`, and appear in `scp` by short name (so the canonical MSAL `scopes:['User.Read']`→`/me` quickstart works and is not rejected as `invalid_scope`). Graph-shaped JSON (`@odata.context`, `value[]`); offset paging via `$top`/`$skiptoken` with `@odata.nextLink` preserving the caller's query params.
  5. **Admin REST API conventions (#11, amends #2 seed/reset).** Unauthenticated (locked). Single admin error envelope `{ error: { code, message, target?, details? } }` with codes `validation_error|not_found|conflict|invalid_reference|internal_error`. Offset pagination (`top`/`skip`, response `{ value, count, top, skip }`) + `search` substring. Secrets are **show-once** (plaintext returned only at creation, hashed at rest, `hint` thereafter); passwords write-only (`hasPassword` boolean). Server generates all entity IDs. Non-null `app_id_uri` is **unique** (409 on duplicate). **Amendment to #2:** `seed.ts` exposes an idempotent skip-existing mode (`INSERT OR IGNORE`) used by `POST /admin/api/seed { force:true }` (non-destructive); `store.reset` **preserves the `tenants` row and active `signing_keys`** when `resetKeys=false` (required by `signing_keys.tenant_id` → `tenants.id` FK) and only empties runtime data tables.
  6. **Portal at `/`, MSAL snippet uses GUID authority (#12).** Portal SPA is served at the single-origin root `/` (overrides draft global-spec §9 `/admin`); consumes only the Admin API + `/health` + discovery. Generated MSAL snippet uses the GUID authority (`<origin>/<tenantId>`) + `knownAuthorities`. Visual styling blocked on `DESIGN.md` (undefined) — ships functional-but-unstyled (same blocker as #6's sign-in page).
  7. **Cross-platform MSAL recipe (#13).** Supported config for all four MSALs: GUID authority + `knownAuthorities`/`validateAuthority=false` + **instance discovery disabled** (offline). Microsoft cloud-host discovery fields stay omitted; `x5c` stays omitted from JWKS unless a platform validator demands it (contingency lands in #3). MSAL.NET/Python are smoke-tested in Iteration 1, fully sampled in Iteration 3. #13 establishes CI provisioning of the .NET SDK + Python runtime.
  8. **Run targets (#14).** Two Iteration-1 targets — `npm start` from source and Docker — share one config/data model. Docker base is Node ≥22.5 (target 24) for `node:sqlite`; a single named volume persists `data/` (DB + cert); container `HEALTHCHECK` hits `/health`. Single-executable is Iteration 2 (#17).
- **Rationale**: Keeps the eight Batch B specs internally consistent and aligned with Batch A's locked token/claim/error/path contracts, and pre-empts the highest-risk MSAL-compat footguns (refresh-rotation security, custom-authority/instance-discovery, issuer matching) while preserving the dev-tool simplicity (auto-consent/auto-grant, unauthenticated admin).
- **Alternatives**: Non-rotating or single-token-revocation refresh (rejected — weaker security, not MSAL-realistic); modeling app-role assignment + Graph permission enforcement in MVP (rejected — large surface for little MVP value, breaks the auto-consent token shape); separate post-logout-URI registry (rejected — reuse registered redirect URIs); adding Microsoft cloud-host discovery fields / `x5c` preemptively (rejected — would point at real hosts / unneeded by MSAL validators).

### Visual identity: Fluent-mimic, distinction carried by an amber "LOCAL EMULATOR" badge
- **Date**: 2026-06-22
- **Context**: The sign-in page (#6) and admin portal (#12) were blocked on `DESIGN.md` being `Status: undefined`; both shipped functional-but-unstyled pending a brand identity. The designer (Murdock) established the identity. The product's whole value is *familiarity* — MSAL developers should feel they're using the real Entra/Azure experience — which is in direct tension with the hard requirement to never impersonate Microsoft and to always read as a local sandbox.
- **Decision**: Commit to a single aesthetic direction — **Fluent-mimic**. White surfaces, restrained Fluent elevation, 4px default corners, a Segoe type voice, Azure communication-blue actions; the sign-in page mirrors the `login.microsoftonline.com` centered-card + account-picker pattern, and the portal mirrors the Azure shell (slim top bar + side nav + dense tables). The single, deliberate signal that this is **not** production is a persistent **amber "LOCAL EMULATOR" badge** (the caution color) in the portal top bar and on the sign-in card, plus a quiet "Not for production use" note. Distinction is carried by that badge + caution signalling, **not** by inventing a separate visual language. No Microsoft logos/trademarks used as our own; our own "Entra Local" wordmark. Captured in `DESIGN.md` (Overview/Positioning/Components — `badge-emulator`), validated `npx @google/design.md lint DESIGN.md` → 0 errors / 0 warnings.
- **Rationale**: Maximum muscle-memory transfer for the target developer while a single high-salience caution element removes any production/official-Microsoft ambiguity — the cheapest, clearest way to reconcile "feels like Entra" with "is obviously a sandbox."
- **Alternatives**: A distinct non-Microsoft visual language (rejected — sacrifices the familiarity that is the product's point); a faint/textual-only disclaimer (rejected — too easy to miss; the caution badge must be unmissable on every screen); replicating Microsoft branding 1:1 (rejected — trademark/impersonation risk, explicitly prohibited).

### Typography: Segoe UI (system) → Selawik fallback, Cascadia Mono for identifiers (license-clean)
- **Date**: 2026-06-22
- **Context**: The Fluent-mimic look depends on the Segoe type voice, but Segoe UI cannot be embedded/redistributed as a webfont. Machine identifiers (GUIDs, client IDs, secrets, tokens, scopes) are pervasive in this tool and must be unambiguous and copyable.
- **Decision**: UI/body/display uses **`"Segoe UI"` referenced from the system stack only** (never embedded), with **Selawik** (Microsoft's OFL, Segoe-metric-compatible font, **self-hosted**) as the web fallback, then `system-ui, -apple-system, sans-serif`; Semibold 600 for titles/labels, Regular 400 for body. Monospace is **Cascadia Mono** (Microsoft, OFL, self-hosted), **required** for every identifier rendered as a copyable mono chip, and for code/MSAL snippets. Type scale + license note captured in `DESIGN.md` Typography.
- **Rationale**: Selawik reproduces Segoe's metrics/feel where Segoe is absent without redistributing a proprietary font, keeping the look license-clean; Cascadia Mono is the natural Microsoft-adjacent, OFL monospace and makes GUID/secret/scope values legible and unmistakably "machine values."
- **Alternatives**: Embedding a Segoe webfont (rejected — license violation); a generic system-font stack with no Selawik fallback (rejected — drifts off-metric on non-Windows, weakening the mimic); a generic mono like Roboto Mono/Consolas-only (rejected — Cascadia is the on-brand OFL choice).

### TLS self-signed cert generation: `selfsigned` package (pure-Node fallback taken)
- **Date**: 2026-06-22
- **Context**: Feature #1 requires HTTPS-by-default with an auto-generated, persisted, stable self-signed cert (CN=localhost, SAN localhost/127.0.0.1/::1, RSA-2048). The spec preferred trying a pure-Node `node:crypto` X.509 self-sign first, with the `selfsigned` npm package as the approved fallback if pure-Node proved impractical.
- **Decision**: Use the `selfsigned` package to mint the self-signed cert/key (persisted under `TLS_CERT_DIR`, regenerated only if absent). `node:crypto` is still used for keypair-adjacent needs and for `X509Certificate` fingerprinting (SHA-256) in stability tests.
- **Rationale**: `node:crypto` can generate keypairs (`generateKeyPairSync`) but has **no API to build/sign an X.509 certificate** — a pure-Node self-sign would require hand-rolling ASN.1/DER encoding, which is brittle and out of scope for #1. `selfsigned` is small, dependency-light, and produces a standard cert with the required SANs; the cert is persisted so the fingerprint is stable across restarts (asserted by tests).
- **Alternatives**: Hand-rolled `node:crypto` ASN.1/DER X.509 self-sign (rejected — brittle, large effort, no upstream API); `node-forge` directly (rejected — `selfsigned` wraps it with a simpler, purpose-built API); shipping a checked-in dev cert (rejected — not per-install unique, and the spec mandates auto-generation).

### Password & secret hashing: scrypt via `node:crypto` (feature #2)
- **Date**: 2026-06-22
- **Context**: Feature #2 (SQLite store) needs to hash user passwords and app client secrets at rest, shared by the deterministic seed and the admin API (#11). The locked driver decision forbids new native deps; `node:sqlite` and `node:crypto` are the only sanctioned primitives. The spec allowed argon2id or scrypt.
- **Decision**: Hash passwords and app secrets with **scrypt** (`node:crypto.scryptSync`, N=16384/r=8/p=1, 16-byte random per-hash salt, 32-byte key) in a self-describing `scrypt$N$r$p$salt$hash` (base64) format; verify in constant time via `timingSafeEqual`. Opaque high-entropy tokens (refresh tokens) are hashed with unsalted **SHA-256**, the digest doubling as the primary key / lookup value. Implemented in `src/store/hashing.ts`.
- **Rationale**: scrypt is a memory-hard KDF built into Node — no native dependency (argon2 packages need native bindings, which conflicts with the single-executable goal #17 and the `node:sqlite` no-native-deps rationale). SHA-256 is sufficient for high-entropy opaque tokens (not low-entropy passwords) and lets the hash serve as a stable PK.
- **Alternatives**: argon2id (rejected — best-in-class but requires a native/WASM dep, against the no-native-deps stance); PBKDF2 (acceptable but not memory-hard); storing tokens with scrypt (rejected — unnecessary cost for high-entropy values and breaks the hash-as-PK lookup).

### HMAC-signed transient `__el_state` carries authorize params across the sign-in POST (#6)
- **Date**: 2026-06-22
- **Context**: The interactive sign-in page POSTs back to `/authorize`; the original OAuth request params (client_id, redirect_uri, scope, state, nonce, PKCE challenge, response_mode, prompt) must survive the round-trip without trusting the browser to not tamper with them, and without server-side per-request storage.
- **Decision**: Serialize the validated `AuthorizeState` to JSON and sign it with HMAC-SHA-256 (`src/identity/authState.ts`, key derived from the signing service / config) into a hidden `__el_state` form field. On submit the signature is verified, then the client_id/redirect_uri are **re-validated against the store** (defense-in-depth) before issuing a code. Presence of `__el_state` distinguishes a sign-in submission from a fresh authorize POST.
- **Rationale**: A signed stateless token avoids server-side login-session storage and is tamper-evident; re-validating the client/redirect on submit means even a forged-but-unsigned-path request can never drive an open redirect. HMAC (not encryption) is sufficient because the params are not secret, only integrity-sensitive.
- **Alternatives**: Server-side transient login-request rows keyed by an opaque id (rejected — extra table + cleanup for no security gain over a signed token); encrypting the state (rejected — params are non-secret, integrity is the only requirement); trusting hidden fields unsigned (rejected — lets a tampered redirect_uri reach code issuance).

### Refresh-token reuse → family revocation via a chain-traversal repository primitive (#7)
- **Date**: 2026-06-22
- **Context**: Feature #7 owns the refresh-token rotation + reuse-detection policy that #5's `redeemRefreshToken` contract deferred. The spec mandates that presenting an already-rotated (revoked) refresh token is a replay that must revoke the **entire rotation chain/family** (ancestors + descendants reachable via `rotated_from`), while a still-active-but-expired token is natural expiry with **no** family revocation, and the whole redemption must be atomic so concurrent redemptions of one token mint exactly one successor.
- **Decision**: Added `refreshTokens.revokeFamily(tokenHash): number` to the #2 refresh-tokens repository (`src/store/repositories/refreshTokens.ts`). It BFS-traverses the rotation chain in **both** directions in one SQLite transaction — backward via each row's `rotated_from` (ancestors) and forward via a new `SELECT token FROM refresh_tokens WHERE rotated_from = ?` statement (descendants) — over a `seen` set, then compare-and-set-revokes every member; returns the count of rows transitioned to revoked. The reuse decision lives in `src/tokens/refresh.ts`: `redeemRefreshToken` checks `row.revoked` **before** expiry (reuse precedence) and, when revoked, calls `revokeFamily(oldHash)` then returns `invalid_grant`; an active-but-expired token returns `invalid_grant` with no family call. Rotation stays atomic via the pre-existing `rotate()` compare-and-set (`UPDATE … SET revoked=1 WHERE token=? AND revoked=0` requiring `changes()==1` before inserting the successor). The `/token` `grant_type=refresh_token` branch (`src/identity/token.ts`) reuses #6's `oauthErrors.ts` and a newly extracted shared `authenticateClient` helper (now used by both the auth-code and refresh handlers), enforces the `client_id`↔token binding via `appId`, embeds the pre-issued rotated token only when the post-narrowing grant still carries `offline_access`, and re-mints the ID token when `openid` is granted. Family revocation is keyed strictly to the presented token's chain — independent sign-ins for the same `(app_id, user_id)` are separate roots and survive.
- **Rationale**: A chain-traversal primitive on the repository (not "revoke all tokens for user+app") is the OAuth 2.1 / MSAL rotation security model: a leaked-then-rotated token kills only its own family, preserving concurrent device/browser sessions. Putting the traversal in the repository keeps it in one transaction with direct SQL access; putting the reuse *decision* in the refresh service keeps the policy composed with `redeemRefreshToken`. Checking `revoked` before `expires_at` gives reuse precedence over expiry as the spec requires.
- **Consequence/flag**: This finalized policy changed one pre-existing #5 unit test (`tokens-service.test.ts` refresh rotation, criterion 8) whose assertion "the rotated successor B still works after replaying A" now contradicts family revocation; it was updated to redeem B *before* the replay (the dedicated #7 reuse integration test asserts B fails *after* the replay). In the concurrency race the loser observes `revoked=1` and triggers `revokeFamily`, so the just-minted successor is also revoked — explicitly allowed by the spec ("the reuse path revokes the family"); exactly one successor row is ever minted.
- **Alternatives**: Revoking all `(app_id, user_id)` tokens on reuse (rejected — kills unrelated concurrent sign-ins, contradicts the spec's chain-keyed family); doing reuse detection in the handler instead of the service (rejected — splits the rotation policy across layers); a recursive SQL CTE for traversal (rejected — the BFS-over-`seen` loop is clearer, terminates on any malformed cycle, and the chain is short in practice).

### #14 Run targets: hardened multi-stage Docker + `npm start` (HEALTHCHECK via node:https, runtime dep-pruning, no config branch)
- **Date**: 2026-06-23
- **Context**: Feature #14 packages the emulator as two run targets — `npm start` from source and a Docker container — sharing the one #1 config/data model. The container had to: build server + portal, run as non-root, persist `data/` (DB + auto-cert) to a volume, pass through all #1 env keys, map its port to the host, and report health — all without curl/wget (absent in `node:*-slim`) and without a target-specific config branch.
- **Decision**: (1) **Multi-stage Dockerfile** — a `build` stage (`npm ci` + `npm run build`) compiles the server and the single-file portal; a separate `deps` stage runs `npm ci --omit=dev` to resolve production-only `node_modules`; the `runtime` stage (`node:24-slim`) copies `dist/`, `portal/dist/`, the prod `node_modules`, and `package.json` only — no React/Vite/test toolchain (image ~361 MB, verified `react`/`vite`/`typescript`/`vitest`/`playwright` absent). (2) **Non-root + writable volume** — copies use `--chown=node:node`, `RUN mkdir -p /app/data && chown node:node /app/data` runs **before** `VOLUME ["/app/data"]` so a fresh named volume inherits `node` ownership and the unprivileged process can write the DB + first-run cert; `USER node` for the runtime. (3) **HEALTHCHECK without curl/wget** — an inline `node -e` client (`https.get` with `rejectUnauthorized:false` against `127.0.0.1:${PORT}/health`, exit 0 only on `200` + `{"status":"ok"}`); `node -e` runs as CommonJS even in this `type:module` package, so `require('https')` works. (4) **Port reachability** — the image sets `ENV HOST=0.0.0.0` (overriding #1's `localhost` default) so `-p` publishing reaches the process, while `PUBLIC_ORIGIN`/`ISSUER` still derive to `https://localhost:8443`. (5) **No config branch** — `DB_PATH`/`TLS_CERT_DIR` are NOT set in the image; both targets default under `./data` (the mount point), so the only difference between targets is where `data/` lives. (6) **`npm start` guard** — a `prestart` (`scripts/check-build.mjs`) aborts with an actionable message if `dist/index.js` or `portal/dist/index.html` is missing. (7) **`.dockerignore`** keeps the build context lean (excludes `node_modules`, `data`, `dist`, `portal/dist`, `.git`, `.github`, docs/specs/memory, test artifacts). (8) Added a `docker` CI job (build → run → wait-healthy → host `/health` 200 → portal at `/` → `-e TENANT_ID` passthrough → cert-fingerprint + admin-row persistence across stop/start) and an in-process `test/integration/run-targets.test.ts` (criteria 1/2/8) so the source target + shared model are proven everywhere in `npm test`.
- **Rationale**: A dedicated `deps` stage yields a reproducible prod-only `node_modules` without an in-place `npm prune` (cleaner layer, no dev artifacts). `--chown` on COPY avoids an expensive recursive chown of `node_modules`. The `node -e` healthcheck removes any dependency on a shell HTTP client present in `-slim` and is TLS-aware against the container's own loopback cert. Keeping `DB_PATH`/`TLS_CERT_DIR` unset in the image is what makes "one model, no branch" literally true and testable (asserted against the Dockerfile text).
- **Alternatives**: In-place `npm prune --omit=dev` in the build stage (rejected — leaves the dev install's layer history; a clean `deps` stage is leaner and clearer); `HEALTHCHECK` via `curl --insecure`/`wget --no-check-certificate` (rejected — neither ships in `node:*-slim`); baking `DB_PATH=/app/data/...` into the image (rejected — would be a target-specific branch; the `./data` default already resolves to the volume under WORKDIR `/app`); `chown -R /app` (rejected — slow over `node_modules`; per-COPY `--chown` is targeted).

- **Date**: 2026-06-23

### #17 Single-executable packaging: Node SEA over `pkg`; esbuild→CJS bundle; lazy asset indirection via `src/runtime/assets.ts`
- **Date**: 2026-06-23
- **Context**: Feature #17 needed a self-contained binary bundling the runtime, portal assets, and DB layer. The roadmap "Open Decision" (Single-exe approach: Node SEA vs `pkg`) had to be resolved before building.
- **Decision**: (1) **Chose Node SEA** over `pkg`. Rationale: persistence uses the built-in `node:sqlite` (NO native bindings), so SEA needs zero native-addon handling — the embedded Node *is* the SQLite engine; SEA is the officially-supported, dependency-light path, while `pkg` is deprecated/unmaintained and lags on newer builtins. (2) **Confirmed only two runtime-read assets** — `portal/dist/index.html` (served by `spaFallback.ts`) and `package.json` (version in `version.ts`); the DB schema/seed are inline SQL compiled into the build (`src/store/db.ts`/`seed.ts`), so there are NO external migration asset files to bundle. Both assets are embedded via SEA `assets` (`sea-config.json`) and read with `sea.getAsset(...)`. (3) **Asset indirection in `src/runtime/assets.ts`** (`isSea()`, `readTextAsset(seaKey, () => URL)`): inside a SEA it returns `sea.getAsset(key,'utf8')`, otherwise it reads the file at the caller's `new URL('…', import.meta.url)`. `version.ts`/`spaFallback.ts` were refactored to go through it WITHOUT changing their non-SEA behaviour, so `npm start`/`dist`/Docker keep their existing filesystem reads. (4) **Bundling with esbuild** (`scripts/build-sea.mjs`): `dist/index.js` + all production npm deps → one CJS file `dist-sea/bundle.cjs` with `node:*` (incl. `node:sqlite`/`node:sea`) external; then `node --experimental-sea-config sea-config.json` → blob → copy `process.execPath` → `postject` inject (`NODE_SEA_BLOB`, sentinel fuse `NODE_SEA_FUSE_fce680ab2cc467b6e072b8b5df1996b2`, `--macho-segment-name NODE_SEA` on darwin). New scripts: `build:sea`, `test:sea`. Tooling (`esbuild`, `postject`) is devDependencies only; `dist-sea/`/`*.exe`/`*.blob` are git/docker/prettier/eslint-ignored.
- **Rationale**: SEA + `node:sqlite` sidesteps the project's biggest packaging risk (native SQLite bindings in a single exe) entirely. Routing both assets through one tiny indirection keeps the SEA branch isolated and leaves every existing run path byte-for-byte unchanged.
- **Consequence/flag**: TWO ESM→CJS-bundling quirks were hit and fixed: (a) esbuild empties `import.meta.url` in CJS output, so `new URL('../package.json', import.meta.url)` evaluated **eagerly** would throw `ERR_INVALID_URL` inside the binary — fixed by making `readTextAsset` take a **lazy `() => URL` thunk** that is only invoked on the non-SEA branch; (b) `createRequire(import.meta.url)` also throws on the empty URL, so SEA detection failed and fell through to the filesystem branch — fixed by using the **in-scope CJS `require`** when present (`typeof require !== 'undefined' ? require : createRequire(import.meta.url)`). Both manifested as a working `/discovery` (no asset) but broken `/health` + `/` (assets) until fixed. The Windows binary is **unsigned** (postject prints a benign "signature seems corrupted" warning because it invalidates Node's Authenticode signature); SEA does not cross-compile (build per OS). The smoke test (`test/integration/sea-packaging.test.ts`) is **availability-gated** (`describe.skip` when the binary is absent / platform unsupported) so the default `npm test` never depends on a SEA build, yet runs the 4 checks when the binary exists. Verified locally on Windows x64 / Node v25.9: exe ≈ 94 MB, `/health` 200 + version `0.1.0`, `/` serves `<div id="root">`, discovery 200.
- **Alternatives**: `pkg` (rejected — deprecated/unmaintained, native-binding pain, lags `node:sqlite`); scattering `sea.isSea()` checks across `version.ts`/`spaFallback.ts` (rejected for one clean indirection); embedding the portal/version as string literals baked into the bundle (rejected — would diverge the SEA build from the on-disk assets and bloat the bundle); `useCodeCache:true`/snapshot in `sea-config.json` (rejected — V8-version coupling/flakiness for marginal startup gain); code-signing the dev binary (out of scope for a dev tool).

### #15 Device Code flow: hashed `device_code` + consonant `user_code`; split RFC/human routes; no `slow_down`, no migration; reuse public Sample SPA
- **Date**: 2026-06-24
- **Context**: Feature #15 (RFC 8628 device authorization grant) needs a `/devicecode` endpoint, a human user-code approval surface, and a `device_code` grant on `/token`. The `device_codes` table already exists (migration-001); the spec had to settle credential formats/storage, the approval-page architecture, polling error semantics, and whether new schema/seed are required.
- **Decision**: (1) **`grant_type` = the RFC URN `urn:ietf:params:oauth:grant-type:device_code`** (the exact value `@azure/msal-node` sends), registered in `token.ts`'s `GRANT_HANDLERS`. (2) **`device_code`** = the shared 256-bit `generateOpaqueToken()`, **stored SHA-256-hashed** in the `device_code` PK column (it's a polled bearer credential, hashed at rest like refresh tokens); **`user_code`** = 8 chars from the ambiguity-reduced consonant charset `BCDFGHJKLMNPQRSTVWXZ` grouped `XXXX-XXXX`, stored plaintext (the human types it), compared case-insensitively. (3) **Split routes**: `POST /devicecode` = RFC JSON device-authorization only (replaces the `501` stub); `GET /devicecode` = the approval page; `POST /devicecode/verify` = the approval form state machine (`lookup`→`signin`→`decide`). No content-type sniffing. (4) **No `slow_down`** is ever emitted — the emulator advertises `interval` and MSAL honors it (documented divergence) — and therefore **no new migration**: the existing table suffices (`resource` recomputed from scopes; no `nonce`/`last_polled_at`). (5) **Single-use by row deletion** on successful redemption; lazy expiry on poll. Polling map: pending→`authorization_pending`, denied→`access_denied`, expired→`expired_token`, unknown/app-mismatch→`invalid_grant` (these three new codes added to the closed `OAuthErrorCode` union). (6) **Approval page reuses #6**: the `el_session` SSO cookie + the `signinPage` account-picker/password form (extended with an `extraHiddenFields` option) + a generalized `createSignedStateSigner<T>()` (the `authState` signer becomes one instantiation) carrying `DeviceApprovalState = { userCode, sid }`; CSRF = SameSite=Lax + live session + signed `sid` re-checked. (7) **Reuse the public Sample SPA** (`cccccccc-0000-0000-0000-000000000001`, `is_confidential=0`) as the device client (and #19's Node CLI sample target) — no new seed app; redirect type is irrelevant to device code.
- **Rationale**: Hashing the polled credential and splitting the machine/human surfaces keeps the security model consistent with existing grants and avoids fragile content negotiation. Skipping `slow_down` enforcement removes a schema column and the only timing-flaky test vector while staying MSAL-compatible. Reusing the existing table, the sign-in UI, and the public SPA minimizes blast radius and keeps the flow deterministic for CI.
- **Alternatives**: short-form `device_code` grant string (rejected — MSAL sends the URN); plaintext `device_code` storage (rejected — inconsistent with refresh-token hashing); overloading `POST /devicecode` for both RFC and form submits (rejected — content-type sniffing); enforcing `slow_down` via a `last_polled_at` column + migration (rejected — flaky + unnecessary for MSAL); a terminal `redeemed` status instead of deleting (rejected — delete gives atomic single-use like `authCodes.consume`); seeding a dedicated native/CLI public app (rejected for MVP — the Sample SPA is already a public client).

### #24 implementation findings: emulator scopes refresh tokens strictly → request API scope at sign-in; a second exposed scope (`access_as_admin`) is required for a deterministic 403
- **Date**: 2026-06-25
- **Context**: Implementing the full-stack #24 sample end-to-end (SPA `…0004` → Express API `…0005`) surfaced two emulator behaviours that the original spec/plan did not account for. Both were discovered by running the real `@azure/msal-browser` Playwright smoke against a live emulator, not by code reading alone.
- **Decision**: (1) **Request the API scope at sign-in, not just OIDC.** `src/tokens/refresh.ts` enforces that a refresh-token redemption's requested scopes are a **subset** of the scopes granted at sign-in (over-broad → `invalid_scope`/AADSTS70011) — unlike production Entra, whose refresh tokens are effectively multi-resource. So the SPA's `loginRequest` is `['openid','profile','offline_access', api://…0005/access_as_user, api://…0005/access_as_admin]`; only then does `acquireTokenSilent` for the API scope succeed. (2) **Add a second exposed scope `access_as_admin`** (`dddddddd-0000-0000-0000-000000000003`) on the API app `…0005`. Because the only way to obtain an `aud=…0005` token is to request an `…0005` resource scope, a single-scope API can never mint a valid-audience token that *lacks* the required scope, making a genuine 403 impossible to test. The smoke mints the 403 token via `acquireTokenSilent({ forceRefresh:true })` for **only** `access_as_admin`, which narrows the (rotated) grant so `scp` omits `access_as_user` while `aud` stays `…0005`. The API (`REQUIRED_SCOPE=access_as_user`) returns 403. (3) Seed contract grows by one row: `app_scopes` 2 → 3; `test/integration/store.test.ts` asserts the API app exposes `['access_as_admin','access_as_user']` (repo lists scopes `ORDER BY value`).
- **Rationale**: Both are additive, in-keeping with the existing seed contract, and require **no protocol/schema change**. Requesting the resource scope at sign-in is a standard, documented MSAL SPA pattern and is the honest way to work with the emulator's stricter (more conservative) refresh-token model. The second scope turns the API's scope-enforcement branch into a CI-verified, deterministic 403 rather than dead code. Verified: the Playwright smoke passes all 200/401/403 assertions; root `lint`/`typecheck`/`build`/`test` (311 tests) stay green.
- **Alternatives**: Use `api://…0005/.default` for the 403 case (rejected — `.default` is never in the granted scope set, so the strict refresh-token check returns `invalid_scope` instead of yielding a token; it cannot produce a 403); relax `refresh.ts` to allow scope broadening on refresh (rejected — would weaken a deliberate emulator security property and is out of scope for a sample); drive the 403 with a hand-forged token (rejected — cannot produce a JWKS-valid signature); drop the 403 assertion from the smoke (rejected — scope enforcement is a core teaching point of the sample and a stated acceptance criterion).

### Local-domains subdomain routing on one shared listener (#26)
- **Date**: 2026-06-25
- **Context**: Entra Local served a single origin and split surfaces by path prefix, unlike real Entra (distinct `login.microsoftonline.com`, `graph.microsoft.com`, portal hosts). The user asked for a faithful local-domains DX with a wildcard cert and the subdomains `login.`/`portal.`/`graph.entra.localhost`, while keeping the project's "one process, one port, one cert" simplicity.
- **Decision**: Default `BASE_DOMAIN=entra.localhost`; derive three advertised origins (`login.`/`portal.`/`graph.<baseDomain>:<port>`, each overridable via `LOGIN_ORIGIN`/`PORTAL_ORIGIN`/`GRAPH_ORIGIN`). All three are served on the **one shared `:8443` HTTPS listener, routed by the `Host` header** (`src/http/hostRouting.ts`): a Fastify server-option `rewriteUrl` re-prefixes `/graph` to root Graph paths on the graph host (so `graph.entra.localhost/v1.0/me` hits the registered `/graph/*` handlers and mirrors real `graph.microsoft.com`), and an `onRequest` guard confines each typed host to its slice (cross-slice path → JSON 404; bare `/` on login/graph → a small JSON descriptor). `localhost`/`127.0.0.1` stays a **backward-compat `compat` host that serves every route**. When the three origins collapse to one host (legacy `PUBLIC_ORIGIN`, or the test harness) routing is disabled and every request is `compat`. A single **self-signed wildcard cert** (`*.entra.localhost` + apex + loopback + `LOCAL_DOMAINS`) covers all hosts and is **regenerated when its SANs no longer cover the configured domain set**. A new `entra-local hosts` CLI (print by default, `--apply` to write, `--remove`) manages the cross-platform hosts file because `*.entra.localhost` does not auto-resolve (notably on Windows). Spec: `specs/2026-06-25_26-local-domains.md` (roadmap #26).
- **Rationale**: Host-header routing over one listener preserves the single-process/single-port/single-cert model while giving a realistic multi-host topology. The collapse guard makes the change fully backward-compatible: the Docker healthcheck, existing tests, samples, and the real-MSAL e2e suite keep running on the loopback origin with no DNS/hosts changes. A wildcard leaf avoids a local CA or per-leaf issuance, and regenerate-on-mismatch means switching on local domains "just works" without a stale-cert name mismatch. Print-by-default `hosts` mirrors the `trust` command and never mutates the hosts file without opt-in elevation.
- **Alternatives**: Separate listeners/ports per surface (rejected — breaks one-port simplicity and the single healthcheck); a reverse proxy in front (rejected — adds an external dependency to a self-contained dev tool); path-prefix-only forever (rejected — does not mirror real Entra host separation the user asked for); per-leaf certs from a generated local CA (rejected — larger change; a single wildcard leaf suffices); auto-editing the hosts file (rejected — elevation-sensitive; kept opt-in via `--apply`); making subdomains the only origin (rejected — would force hosts edits on every existing user and break multi-surface samples that call both login and `/graph`).

- **Date**: 2026-07-06
- **Context**: Local domains (#26) made the `login.`/`portal.`/`graph.entra.localhost` subdomains the single default advertised origin for **every** run target. That silently broke localhost-based flows that don't set `PUBLIC_ORIGIN`: the sample CI smokes, the checked-in sample docker-composes, and any `docker run` — because a container cannot make `*.entra.localhost` resolve on the host, MSAL is redirected to non-resolving subdomains and localhost-configured resource APIs reject the login-subdomain `iss`. A container also can't usefully run `hosts --apply` (it edits the container's hosts file, not the host's). ### Response Mode form_post: RFC 8693 support + security-preferred default for new samples
- **Date**: 2026-07-08
- **Context**: Users integrating MSAL.NET and other server-side frameworks reported that Microsoft.Identity.Web defaults to `response_mode=form_post` for security (hides auth parameters from browser history/logs), but entra-local only supported `query` and `fragment`. User feedback confirmed form_post is an official Entra ID capability and requested support for .NET samples to default to it.
- **Decision**: Implement RFC 8693 Form Post Response Mode in the authorization endpoint (#6 feature). The `/authorize` endpoint now accepts `response_mode=form_post` alongside `query` and `fragment`. When form_post is used, the endpoint returns a `200` HTML response containing a hidden form with the authorization code and state; JavaScript auto-submits the form via POST to the `redirect_uri`, keeping auth parameters out of browser history and proxy logs. Discovery document updated to advertise `["query", "fragment", "form_post"]`. Error responses revert to query mode per RFC 8693 (form_post only applies to successful authorization). New .NET console sample (`samples/dotnet-console/`) defaults to form_post when acquiring tokens interactively, demonstrating the security-preferred flow for server-side applications.
- **Rationale**: form_post is a published RFC standard (RFC 8693) actively supported by Entra ID. It improves security by hiding authorization parameters from browser history, HTTP logs, and proxy records — important for production deployments. Server-side frameworks (MSAL.NET, OAuth2 Proxy) default to it for this reason. The implementation is straightforward (HTML form + auto-submit) and fully backward-compatible (query/fragment users unaffected). Making it the default for new server-side samples (while SPA samples keep fragment) follows security best practices.
- **Alternatives**: Keep query-only (simplest but less secure, contradicts Microsoft's defaults); make form_post mandatory (breaking change for existing SPA samples that use fragment); require explicit opt-in per-app (viable but less secure by default); use fragment for all (better than query, but still visible in browser history and proxy logs; not as secure as form_post for server-side apps).

- **Date**: 2026-07-08

Ownership rule mirrors Entra: **ID-token** optional/group claims come from the **client** app; **access-token** claims come from the **resource/API** app resolved from the requested `api://` audience (never the client, unless the client *is* the audience). Unsupported configured claims are preserved but never emitted, and the endpoint logs a warning. Group claims emit stable local group IDs; when the membership count exceeds the limit (per-app override, else config `groupOverageLimit`, default 200; env `GROUP_OVERAGE_LIMIT`) the token instead carries `_claim_names`/`_claim_sources` pointing at `GET /graph/v1.0/me/memberOf`. Added Graph `/me/memberOf` + `/users/:id/memberOf`, admin routes (`GET /admin/api/token-configuration/supported-claims`, `POST /admin/api/apps/:id/token-preview`, plus the three fields on `PATCH /admin/api/apps/:id`), a portal **Token configuration** card (optional-claim editors, group selector, overage limit, decoded token preview), and seeded demo apps `local-web-client` (`…0006`, ID-token claims) + `local-api` (`…0007`, access-token claims) with a low overage limit (3) so a 4-group user (Alice) overflows and a 2-group user (Bob) does not.
- **Rationale**: A single resolver shared by issuance and preview guarantees the portal preview cannot drift from real tokens. Preserving unsupported claims (rather than rejecting them) keeps stored configuration forward-compatible as Entra Local adds claims. Resolving access-token claims from the resource app is the behaviour developers must test for realistic MSAL/OAuth flows. Nullable/defaulted migration columns mean existing app registrations need no data migration.
- **Alternatives**: Emit group display names or SAM names (rejected — Entra emits IDs by default; names come from Graph); apply the client's access-token config to any audience (rejected — incorrect vs Entra; would mislead API-side testing); enable overage only when a limit is set (rejected — a sensible default of 200 matches Entra and keeps behaviour predictable); duplicate claim logic in the preview endpoint (rejected — would let the preview diverge from issuance).
